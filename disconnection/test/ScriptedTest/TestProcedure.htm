Planned Disconnect Defense (ADRT UC#7)

Version 1.0 – 4/17/2003

Test Procedure

 

Summary

 

These tests stimulate the Planned Disconnect Defense by manipulating its disconnection servlet and observing (as events) playbook induced changes to operating modes.  The test is successful if when a specified defense condition is set, that the corresponding operating modes are changed.  This document describes how to set up and run the tests.

 

Requirements (one Management Node & 1 or more Managed Nodes)

 

Note: The release as packaged supports the management of two nodes named NodeA and NodeB.  Customization is required to support nodes with other names.  Specifically, bold-italicized items below must be customized for the particular Managed Nodes that will be allowed to disconnect per the Defense.  The reason for this requirement is the experimental use of the Cougaar Adaptivity Engine in a context other that for which it was designed.  Specific reasons are discussed in the document PlannedDisconnect-Design.

 

In the  Management Node - named ManagementAgent

 

plugin = org.cougaar.core.adaptivity.OperatingModeServiceProvider

plugin = org.cougaar.core.adaptivity.OperatingModePolicyManager

plugin = org.cougaar.core.adaptivity.AdaptivityEngine

plugin = org.cougaar.core.adaptivity.PlaybookManager(CommunityLevel_playbook.txt)

plugin = org.cougaar.core.adaptivity.ConditionServiceProvider

 

plugin = org.cougaar.tools.robustness.disconnection.PlannedDisconnectManagerPlugin(NodeA, NodeB, ...)

plugin = org.cougaar.tools.robustness.disconnection.test.disconnection.DefenseOperatingModeChangeEvents

 

Inclusion of Community_playbook.txt

 

Note: In the above, the names of the Managed Nodes must be supplied as parameters to the PlannedDisconnectManagerPlugin.

 

 

In each Managed Node - named NodeA, NodeB, ...   (arbitrary nodes in the society)

 

plugin = org.cougaar.core.adaptivity.OperatingModeServiceProvider

plugin = org.cougaar.core.adaptivity.OperatingModePolicyManager

plugin = org.cougaar.core.adaptivity.PlaybookManager(NULL_playbook.txt)

plugin = org.cougaar.core.adaptivity.ConditionServiceProvider

plugin = org.cougaar.tools.robustness.deconfliction.RemoteDefenseConditionMgrPlugin(ManagementAgent)

plugin = org.cougaar.tools.robustness.disconnection.test.disconnection.PlannedDisconnectServletTestNodePlugin

plugin = org.cougaar.tools.robustness.disconnection.test.disconnection.DisconnectionServlet

plugin = org.cougaar.tools.robustness.disconnection.test.disconnection.DefenseOperatingModeChangeEvents

 

 Inclusion of NULL_playbook.txt

 

 

Customization of the Community_playbook.txt

 

Because the AE does not support parameterization, a separate set of plays is required for each Managed Node.  If different nodes are to be supported, Community_playbook.txt should contain a set of rules looking like those in the file PlaybookTemplate (with the name of the Managed Node replacing NodeFOO) for each managed node.

 

 

Test Description

 

1) Start the ManagementAgent and all Managed Nodes.

 

 

2) Disconnect a Node.

 

Use the provided Disconnection servlet (for some Managed Node – e.g., NodeFOO) to:

 

·         set the condition "UnscheduledReconnectTime" to T > 0.0 to indicate that the Node is disconnecting for time T

 

This can also be done from the test harness by generating the following HTML:

 

·         XXXX

 

This will cause operating modes to be set for both the Managed Node and the ManagementAgent.  These will be visible as EventService Events.

 

The following Events will be generated by operating modes set on the Node Agent:

 

·         DefenseOperatingMode:PlannedDisconnect.UnscheduledDisconnection.Node.NodeFOO=TRUE

·         DefenseOperatingMode:PlannedDisconnect.UnscheduledReconnectTime.Node.NodeFOO=<T>

·         DefenseOperatingMode:PlannedDisconnect.NodeDefense.Node.NodeFOO=TRUE

 

The following Events will be generated by operating modes set on the Management Agent:

 

·         DefenseOperatingMode:PlannedDisconnect.Applicable.Node.NodeFOO=TRUE

·         DefenseOperatingMode:PlannedDisconnect.Defense.Node.NodeFOO=ENABLED

·         DefenseOperatingMode:PlannedDisconnect.Monitoring.Node.NodeFOO=ENABLED

·         DefenseOperatingMode:PlannedDisconnect.ManagerDefense.Node.NodeFOO=TRUE

·         DefenseOperatingMode:PlannedDisconnect.ManagerMonitoring.Node.NodeFOO=TRUE

 

3) Reconnect a previously disconnected node.

 

Use the provided Disconnection servlet (for some Managed Node – e.g., NodeFOO) to:

 

·         set the condition "UnscheduledReconnectTime" to T = 0.0 to indicate that the Node is reconnecting.

    

This can also be done from the test harness by generating the following HTML:

 

·         XXXX

 

 

The following Events will be generated by operating modes set on the Node Agent:

 

·         DefenseOperatingMode:PlannedDisconnect.UnscheduledDisconnection.Node.NodeFOO=FALSE

·         DefenseOperatingMode:PlannedDisconnect.UnscheduledReconnectTime.Node.NodeFOO=0.0

·         DefenseOperatingMode:PlannedDisconnect.NodeDefense.Node.NodeFOO=FALSE

 

The following Events will be generated by operating modes set on the Management Agent:

 

·         DefenseOperatingMode:PlannedDisconnect.Applicable.Node.NodeFOO=FALSE

·         DefenseOperatingMode:PlannedDisconnect.Defense.Node.NodeFOO=DISABLED

·         DefenseOperatingMode:PlannedDisconnect.Monitoring.Node.NodeFOO=DISABLED

·         DefenseOperatingMode:PlannedDisconnect.ManagerDefense.Node.NodeFOO=FALSE

·         DefenseOperatingMode:PlannedDisconnect.ManagerMonitoring.Node.NodeFOO=FALSE

 

Notes:

·         Steps #2-3 can be performed for any of the Managed Nodes.

·         For any given Managed Node, a sequence of (disconnect – reconnect)* will produce the Events above, regardless of disconnects & reconnects on other managed nodes.

·         Operating modes will be set properly for a given managed node if a disconnected node attempts to disconnect again or an already connected node attempts to reconnect (e.g., disconnect – disconnect – reconnect – reconnect).  However, because operating modes are not “changed” if they are already the right value (e.g., set from TRUE to TRUE), the set of events published will not be the same as above (it will be a subset, with values “carrying over”).
