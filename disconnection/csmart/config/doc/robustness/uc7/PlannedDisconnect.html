Planned Disconnect Defense

ARDT UC#7

David Wells – OBJS

9/2/2003

 

 

Purpose of the Defense

The Disconnect Defense allows Nodes to declare that they are intentionally disconnecting from the network for some length of time.  During this time, being uncommunicative is their “normal” state and thus should not trigger other defense actions such as restart or repeated attempts to communicate via alternate channels or protocols.  In other words, defenses such as Restart and MsgLog should not attempt to “fix” a non-existent problem. 

 

Interaction with the Coordinator

The Disconnect Defense allows nodes to request to disconnect or to reconnect.  Such a request is generally granted, but until the Disconnect Defense has been enabled, the node cannot legitimately disconnect and expect no consequences.   Attempts to disconnect the node containing the robustness community’s Robustness Agent are rejected, as the Robustness Agent cannot be allowed to remove itself from the community.  When nodes disconnect, they provide an expected duration for the disconnect; if they do not reconnect within that time, they are declared AWOL and the node and all its agents are flagged as no longer legitimately disconnected, making them eligible for other defensive action (including restart).

 

While disconnection occurs at the Node level, the Coordinator and all other Defenses reason about the world at the level of Agents.  Thus, when a Node wishes to signal its intention to Disconnect, the Defense does so by informing the Coordinator in terms of the Agents that will be disconnected as a result. 

 

The Disconnect Defense adheres to the ADRT UC#9 Deconfliction API.  As such, it publishes the Condition and OperatingMode objects for each managed Agent for use by the Coordinator.  See the Deconfliction documentation for more information on the purpose of these objects.  The specific classes are:

 

·       DisconnectApplicabilityCondition – a subclass of DefenseApplicabilityBinaryCondition – TRUE if the Agent is planning to disconnect; FALSE if the Agent is not planning to disconnect

·       DisconnectDefenseEnabler – a subclass of DefenseEnablingOperatingMode – Defense is ENABLED/DISABLED from acting on the asset. This means that it is OK for the Node to disconnect and that none of its agents will be restarted or otherwise affected by the disconnect.

·       DisconnectMonitoringEnabler – a subclass of MonitoringEnablingOperatingMode  – Monitoring is ENABLED/DISABLED for the asset. This has no real effect.

 

Interactions with the Outside World

When a node has been granted permission to disconnect, the following Cougaar event is generated by the Node for the node and for each agent on the node:

 

·       <NODE> plans to Disconnect for <TIME> sec

·       <AGENT> plans to Disconnect for <TIME> sec

 

When a node has been acknowledged as being reconnected, the following Cougaar event is generated by the Node for the node and each local agent:

 

·       <NODE> has Reconnected

·       <AGENT> has Reconnected

 

If a node fails to reconnect within the allotted time, the following Cougaar event is generated by the Disconnect Manager for the node and all its agents:

 

·       <NODE> is no longer legitimately Disconnected

·       <AGENT> is no longer legitimately Disconnected

 

If an attempt is made to disconnect the node containing the robustness community’s Robustness Agent, the following Cougaar event is generated by the Node:

 

·       Not allowed to Disconnect RobustnessAgent on <NODE>

 

Structure of the Defense

Internally, the Defense has a presence in:

 

·       Nodes

·       Agents

·       A Distinguished Agent containing the Disconnect Manager for a Robustness Community.  This Distinguished Agent probably, but does not necessarily, reside in the RobustnessAgent for the Robustness Community.

 

Disconnect Defense Presence in Nodes

A Node is the logic unit of disconnect.  A DisconnectNodePlugin provides the basic capability for a node, supported by a DisconnectServlet though which disconnect requests can be made.

 

The DisconnectNodePlugin:

 

·       Locates the Robustness Agent using the Community Service

·       Creates a LocalReconnectTimeCondition used by the servlet for signaling disconnect and reconnect requests

·       Subscribes to adds and removes of AgentExists objects published by the DisconnectAgentPlugins to keep a current list of agents on the node

·       Creates and destroys ReconnectTimeCondition, DisconnectDefenseAgentEnabler, DisconnectMonitoringAgentEnabler objects for each of the local agents and makes these available to the DisconnectManagerPlugin

·       Subscribes to changes in the LocalReconnectTimeCondition object used by the servlet to signal disconnect/reconnect requests

·       Maps the node-level disconnect/reconnect request to all of the local agents and sets the ReconnectTimeCondition for all the agents based on the value passed from the servlet in the LocalReconnectTimeCondition

·       Subscribes to DisconnectDefenseAgentEnabler and DisconnectMonitoringAgentEnabler to determine the status of disconnect & reconnect requests, and publishes Cougaar events giving the status

 

The DisconnectServlet:

 

·       Provides a GUI and an API through which disconnect and reconnect requests can be made.  A disconnect request is done by setting the reconnect time to a positive number; a reconnect request is done by setting the reconnect time to zero.

·       Does some rudimentary checking to ensure that the time provided actually is a non-negative number and that the Robustness Agent has been located.

 

Disconnect Defense Presence in Agents

Each agent contains a DisconnectAgentPlugin whose sole responsibility is to notify the DisconnectNodePlugin of the presence of the agent on the node.  It does this by:

 

·       publishAdd() an AgentExists object when the agent is loaded

·       publishRemove() the AgentExists object when the agent is unloaded

 

Disconnect Defense Disconnect Manager

The Disconnect Manager is responsible for monitoring whether a Node has reconnected when it said it would and handling communications with the Defense Coordinator.  The Disconnect Manager probably resides in the Robustness Agent of the Robustness Community, but this is not a requirement.

 

The Disconnect Manager contains a DisconnectManagerPlugin, whose responsibilities are:

 

·       Locate the Robustness Agent using the Community Service

·       Publish DisconnectApplicabilityCondition, DisconnectDefenseEnabler, and DisconnectMonitoringEnabler objects for each agent that has registered (see DisconnectNodePlugin) and make these available to the Defense Coordinator residing in the RobustnessAgent.

·       Manage alarms to keep track of whether nodes have reconnected on time.  If they have, no action is taken.  If they haven’t, then the Node and all its local agents are declared to be no longer requesting to be disconnected by setting their DisconnectApplicabilityConditions to FALSE.  A Cougaar event is emitted informing the world of the failure to reconnect.

·       Transfer condition and operating mode values between the Coordinator (which uses one set of objects) and the nodes (which use a different set).  This dual set of objects is required because the Disconnect Manager must be able to mediate the condition objects seen by the Coordinator to allow timing out of tardy Nodes.

 

Installation & Use

The Defense resides in the package org.cougaar.tools.robustness.disconnect. 

 

Agents must contain the following plugIns:

 

[ Plugins ]

# Relayable Conditions & OpModes

plugin = org.cougaar.core.adaptivity.OperatingModeServiceProvider

plugin = org.cougaar.core.adaptivity.OperatingModePolicyManager

plugin = org.cougaar.core.adaptivity.ConditionServiceProvider

 

# Disconnect

plugin = org.cougaar.tools.robustness.disconnect.DisconnectAgentPlugin

 

The first three allow relayable conditions and operating modes.  The fourth is the disconnect logic for the agent. 

 

In addition, if agent mobility is to be exploited, the following optional plugIns are required:

 

# Optional plugIns for agent mobility

plugin = org.cougaar.core.mobility.servlet.MoveAgentServlet

plugin = org.cougaar.core.mobility.service.RedirectMovePlugin

 

 

 

 

Nodes must contain the following plugIns:

 

[ Plugins ]

# Relayable Conditions & OpModes

plugin = org.cougaar.core.adaptivity.OperatingModeServiceProvider

plugin = org.cougaar.core.adaptivity.OperatingModePolicyManager

plugin = org.cougaar.core.adaptivity.ConditionServiceProvider

 

# Disconnect

plugin = org.cougaar.tools.robustness.disconnect.DisconnectNodePlugin

plugin = org.cougaar.tools.robustness.disconnect.DisconnectServlet

 

# Community Service

plugin = org.cougaar.community.CommunityPlugin

plugin = org.cougaar.community.util.CommunityViewerServlet

plugin = org.cougaar.tools.robustness.disconnection.RobustnessManagerFinderPlugin

 

The first three allow relayable conditions and operating modes.  The fourth and fifth are the disconnect logic for the agent.  The last three allow finding the RobustnessManager.

 

 

 

The Disconnect Manager must contain the following plugIns:

 

[ Plugins ]

# Relayable Conditions & OpModes

plugin = org.cougaar.core.adaptivity.OperatingModeServiceProvider

plugin = org.cougaar.core.adaptivity.OperatingModePolicyManager

plugin = org.cougaar.core.adaptivity.ConditionServiceProvider

 

# Disconnect

plugin = org.cougaar.tools.robustness.disconnect.DisconnectManagerPlugin

 

# Community Service

plugin = org.cougaar.community.CommunityPlugin

plugin = org.cougaar.community.util.CommunityViewerServlet

plugin = org.cougaar.tools.robustness.disconnection.RobustnessManagerFinderPlugin

 

The first three allow relayable conditions and operating modes.  The fourth is the disconnect logic for the agent.  The last three allow finding the RobustnessManager.

 

In addition, if agent mobility is to be exploited, the following optional plugIns are required:

 

# Optional plugIns for agent mobility

plugin = org.cougaar.core.mobility.servlet.MoveAgentServlet

plugin = org.cougaar.core.mobility.service.RedirectMovePlugin

 

Finally, Disconnect Defense must run in a community with a Defense Coordinator residing in the Robustness Manager.   Strictly speaking, the Coordinator is not part of the Disconnect Defense, but must be present.  In the current release, the Disconnect Manager must reside in the  Robustness Manager (this restriction will eventually be lifted).   To support this, the Disconnect Manager must reside in a Robustness Manager containing the following plugin to support Deconfliction:

 

# Deconfliction

plugin = org.cougaar.tools.robustness.deconfliction.CoordinatorPlugin(10)

 

The parameter allows setting a delay for another Defense and is explained in the documentation for the Deconfliction.
