ACME Automation for Adaptive Robustness Use Case #7
Planned Disconnect Experiment
OBJS' Disconnection module

OBJS' Disconnection defense for ARUC7, along with MIC's Sledgehammer for ARUC1 and OBJS' MsgLog for ARUC2, is coordinated by OBJS' Coordinator module.

These instructions describe the ACME automation for the Disconnection experiment.

Overlays:
Planned Disconnect requires the following overlays be loaded:
Robustness Telcordia believability/coordinator HEAD B11_2
Robustness Objs Coordinator HEAD B11_2
Robustness Objs sensors HEAD B11_2
Robustness Objs disconnection HEAD B11_2
Robustness Objs Common HEAD B11_2
Robustness Boeing ExNihilo HEAD B11_2
Robustness MIC Mgmt Agent HEAD B11_2
Robustness Robustness common HEAD B11_2

Note: Planned Disconnect itself is implemented by OBJS' Disconnection module, but it depends on OBJS' Coordinator, which depends on Telcordia's Believability and MIC's Mgmt Agent, which in turn depends on OBJS' Sensors & Common, Boeing's ExNihilo & Robustness Common.

ACME Actions and States
Planned Disconnection requires Acme actions and states defined in the following file:

$CIP/csmart/config/lib/robustness/uc7/disconnection.rb

They are documented at the end of this file.

Parameterized Experiment Definition

Include the following scriptlet in a run script to add a Disconnection experiment to a run:

$CIP/csmart/lib/robustness/objs/planned_disconnect.rb

$CIP/csmart/scripts/definitions/robustness/objs/ARUC7-UA-Stop-PostStage1.rb

  - script: $CIP/csmart/lib/robustness/objs/planned_disconnect.rb
    parameters:

      - location: during_stage_1

ACME Actions and States Detail
$CIP/csmart/config/lib/robustness/uc7/disconnection.rb defines the ACME Actions and States that the above script uses.

Note: All action & states take an optional parameter (0, 1, 2, 3) to allow the verbosity of messaging to the acme console to be set. 
Default is 0 which prints no messages, 1 adds error messages, 2 adds info messages, and 3 adds debug messages. 
We suggest a setting of 2 during testing and 1 otherwise.

# This action performs the bulk of the work.  It starts the PlannedDisconnect,
# confirms Permission to Disconnect, performs the actual network disconnect
# and reconnect, initiates the Request to Reconnect, confirms the Permission
# to Reconnect, reports the final status and handles any deviations that occur
# along the way. It runs in a separate thread so as not to interfere with the
# progress of the run script, especially detection of quiescence, and sets a
# a flag when complete.
do_action "PlannedDisconnect", ["UA-FSB-A-NODE", "UA-FSB-C-NODE"], 12.minutes, 8.minutes, 2

Note
If the actual disconnect time (plus four minutes) is greater than the
planned disconnect time, then the following messages will be emitted:

UA-FSB-A-NODE is Tardy and is no longer legitimately Disconnected
UA-FSB-C-NODE is Tardy and is no longer legitimately Disconnected

and the ARManager should restart the agents on these nodes on other nodes.
When the actual disconnect concludes, the disconnected nodes and agents
will return, resulting in duplicate agents. 

In the past, Cougaar could not tolerate duplicate agents and the run would
not complete successfully.  Now, under 11.2, this problem is reported to be
resolved, and the Tardy agents will be shunned.
For further information configuring these defenses, see https://cvs.ultralog.net/ultralog/UltraLogRobustness.
